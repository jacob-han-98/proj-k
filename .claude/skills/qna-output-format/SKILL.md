# QnA 답변 포맷 & 퀄리티 가이드

이 스킬은 **지식 베이스 기반 QnA 에이전트**(예: Project K agent-sdk-poc, qna-poc)가 답할 때의 **표현 규칙**이다. "무엇을 답할까"는 에이전트 프롬프트(CLAUDE.md)가 정하고, "어떻게 보여줄까"는 이 스킬이 정한다.

읽는 순서:
1. 답변 구조 — 4단 기본형
2. 근거 인용 규칙
3. 원문 보존 규칙 (수치·공식·표·다이어그램)
4. 진행 상태 표시 (스트리밍 시)
5. 모름·불확실 처리
6. 금지 사항

---

## 1) 답변 구조 — 기본 4단

**모든 답변은 다음 네 블록을 이 순서로 구성한다.** 질문이 간단하면 블록을 비워도 되지만 순서는 지킨다.

```markdown
## 결론
<1~3문장. 사용자가 질문한 것에 대한 직접 답.>

## 근거
<원문 발췌·수치·공식·표. 각 주장 뒤 (출처: <경로> § <섹션>).>

## 관련 시스템 / 맥락
<교차 참조·연쇄 영향·디자인 의도. 없으면 생략.>

## 더 볼만한 곳
<기획자가 주제를 확장할 때 읽을 인접 문서/시스템 2~4개.>
```

— 특별 케이스:
- **목록형 질문**("어떤 게 있나?"): 결론=개수, 근거=분류된 목록, 관련=교차 참조.
- **수치형 질문**("몇 초인가?"): 결론에 값·단위·내부 상수명 포함, 근거에 원문 표.
- **개념·정의형**("X란?"): 결론에 한 문장 정의, 근거에 핵심 용어·특징.

## 2) 근거 인용

**모든 사실 주장 뒤 출처를 표기한다.**

### 형식

```
(출처: <경로 또는 워크북/시트> § <섹션 제목>)
```

- 경로는 프로젝트 공통 상대 경로 (예: `packages/xlsx-extractor/output/7_System/PK_HUD 시스템/HUD_전투/_final/content.md`).
- 섹션은 실제 헤딩 문구 그대로. 예: `§ ② 공식 - 6) 치명타 적중 / 치명타 저항`.
- 섹션 제목에 괄호(`(7)` 등)가 있어도 그대로 유지. 파서가 괄호 balance를 지원한다.
- **여러 출처**는 한 문장이 여러 곳에 근거할 때 `(출처: A; 출처: B)` 로 나열.

### 배치

- 표·수식 **직후** 한 줄 아래.
- 단락 **끝**에 한 번.
- 답변 맨 마지막에 몰아서 쓰지 않는다 (근거 ↔ 주장 대응이 애매해짐).

## 3) 원문 보존

| 종류 | 규칙 |
|------|------|
| 수치/상수 | 원문 표기 그대로 (예: `999,999`, `MetaSwitchCoolTime = 3`). 단위 변환 금지. |
| 공식 | 한 줄 수식 블록: `` ```수식 ``` `` 또는 Markdown 인용. 기호를 임의로 바꾸지 않는다. |
| 표 | 원본 열 순서·머리글 유지. 필요 시 일부 컬럼만 발췌하되 원본 표기 유지. |
| Mermaid/플로우 | 원본 그대로 코드블록(` ```mermaid `)에 넣는다. 단순화 금지. |
| 이미지 | 가능하면 원문 설명문을 그대로 옮기고, 이미지 자체는 경로 언급만. |
| 영문 식별자 | CamelCase, Enum 값 등은 원문 그대로 (예: `CharacterSkillClass.SkillType = Attack`). |

## 4) 진행 상태 표시 (스트리밍)

에이전트가 툴 호출 시 사용자에게 다음 이모지 라벨을 붙인다.

| 단계 | 이모지 | 예시 메시지 |
|------|--------|-------------|
| 분석/계획 | 🧠 | "질문을 분석하고 있습니다..." |
| 내부 사고 | 💭 | "💭 ..." + 첫 80자 |
| 탐색 (Grep) | 🔎 | "`키워드` 검색 중 (path)" |
| 탐색 (Glob) | 📂 | "패턴 매칭: `pattern`" |
| 읽기 (Read) | 📖 | "`basename` 읽는 중" |
| 관계 조회 (KG) | 🔗 | "시스템 관계 조회 ..." |
| 용어 조회 | 🔤 | "용어 정규화 ..." |
| 종합 | ✨ | "종합하고 있습니다..." |
| 완료 | ✅ | 응답 완료 |
| 오류 | ❌ | 오류 메시지 |

- 각 이벤트는 한 줄이며 최대 120자.
- 원문 경로/키워드는 백틱(`)으로 감싼다.

## 5) 모름·불확실

- 문서에서 확인되지 않으면 **"문서에서 확인되지 않습니다."** 명시. 이어서 **어떤 경로를 탐색했는지** 적는다.
- 추측으로 채우지 않는다. 대신 "가능성 있는 인접 시스템: ..." 형태로 다음 탐색 힌트를 제공한다.
- 원본이 TBD/작성 예정인 경우 그대로 전달 (해석·보완 금지).

예:
```
문서에서 확인되지 않습니다. 아래 경로를 탐색했으나 수치가 명시되어 있지 않습니다.
- `PK_변신 및 스킬 시스템/변신.md` — 쿨타임 존재만 언급
- `PK_HUD 시스템/HUD_전투.md` — 별도 상수 테이블 없음

관련 가능성: `ContentSetting.xlsx` 상수 테이블, `PK_밸런싱` 문서.
```

## 6) 금지 사항

1. **추측 답변** — 문서에 없는 값을 만들어 채우지 않는다.
2. **요약 대체 금지** — 핵심 수치·공식을 "적당히"로 요약하지 않는다. 필요 시 원문 인용.
3. **출처 누락** — 사실 주장은 반드시 인용. 인용 없는 주장은 "내부 지식에 기반한 해석입니다"로 명시.
4. **마크다운 남용** — `**굵게**` 는 핵심 결론/수치에만. 문단 전체 굵게 금지.
5. **장식 이모지** — 메시지 내용 이모지(🎉, 💯)는 쓰지 않는다. 진행 상태 라벨 이외에 사용 금지.
6. **상급 메타 표현** — "포괄적으로", "종합적으로", "탁월하게" 같은 자화자찬 표현 금지.

세부 규칙: https://… 또는 프로젝트 내 `.claude/skills/qna-output-format/SKILL.md`.

(핵심만 요약)
- 결론→근거→관련→더 볼만한 곳 4단
- 모든 주장 뒤 `(출처: <경로> § <섹션>)`
- 수치/공식/표는 원문 보존, 모름은 "확인되지 않습니다"
```
