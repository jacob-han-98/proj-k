# QnA 답변 포맷 & 퀄리티 가이드

이 스킬은 **지식 베이스 기반 QnA 에이전트**(예: Project K agent-sdk-poc, qna-poc)가 답할 때의 **표현 규칙**이다. "무엇을 답할까"는 에이전트 프롬프트(CLAUDE.md)가 정하고, "어떻게 보여줄까"는 이 스킬이 정한다.

읽는 순서:
1. 답변 구조 — 4단 기본형
2. 근거 인용 규칙
3. 원문 보존 규칙 (수치·공식·표·다이어그램)
4. 진행 상태 표시 (스트리밍 시)
5. 모름·불확실 처리
6. 금지 사항

---

## 1) 답변 구조 — 기본 4단

**모든 답변은 다음 네 블록을 이 순서로 구성한다.** 질문이 간단하면 블록을 비워도 되지만 순서는 지킨다.

```markdown
## 결론
<1~3문장. 사용자가 질문한 것에 대한 직접 답.>

## 근거
<원문 발췌·수치·공식·표. 각 주장 뒤 (출처: <경로> § <섹션>).>

## 관련 시스템 / 맥락
<교차 참조·연쇄 영향·디자인 의도. 없으면 생략.>

## 더 볼만한 곳
<기획자가 주제를 확장할 때 읽을 인접 문서/시스템 2~4개.>
```

— 특별 케이스:
- **목록형 질문**("어떤 게 있나?"): 결론=개수, 근거=분류된 목록, 관련=교차 참조.
- **수치형 질문**("몇 초인가?"): 결론에 값·단위·내부 상수명 포함, 근거에 원문 표.
- **개념·정의형**("X란?"): 결론에 한 문장 정의, 근거에 핵심 용어·특징.

## 2) 근거 인용

**모든 사실 주장 뒤 출처를 표기한다.**

### 형식 (답변 본문 내 인라인)

```
(출처: <경로> § <섹션 제목>)
```

- 경로는 **LLM이 실제로 Read한 내부 파일 경로**를 그대로 적는다 (예: `../xlsx-extractor/output/7_System/PK_HUD 시스템/HUD_전투/_final/content.md`). 파서가 이 경로를 원본 식별자로 변환한다.
- 섹션은 실제 헤딩 문구 그대로. 예: `§ ② 공식 - 6) 치명타 적중 / 치명타 저항`.
- 섹션 제목에 괄호(`(7)` 등)가 있어도 그대로 유지 (파서가 괄호 balance 지원).
- **축약 금지** — `(출처: 위와 동일)` / `(상동)` / `(같음)` 사용 X. 매번 전체 경로 반복.

### 사용자 표시 규칙 (중요 — 서버·UI 가 자동 변환)

**내부 가공 경로는 사용자에게 노출하지 않는다.** `packages/xlsx-extractor/output/...` 같은 내부 경로는 **로그·디버그 전용**이며, 사용자 UI 에는 아래 원본 표기로 변환되어 보여야 한다:

| 소스 | 내부 경로 (LLM 인용) | 사용자 표시 (origin_label) | 클릭 링크 (origin_url) |
|------|---------------------|---------------------------|-------------------------|
| Excel (xlsx-extractor) | `packages/xlsx-extractor/output/.../워크북명/시트명/_final/content.md` | `워크북명.xlsx / 시트명 시트` | (없음 — xlsx 는 로컬 파일) |
| Confluence | `packages/confluence-downloader/output/공간/…/페이지/content.md` | `Confluence / 공간 / … / 페이지` | `https://bighitcorp.atlassian.net/wiki/pages/viewpage.action?pageId=<id>` (manifest 매핑) |

구현: `storage._path_to_source_meta()` 가 위 매핑을 수행하여 `{origin_label, origin_url}` 필드를 생성. Confluence pageId 는 `packages/confluence-downloader/output/_manifest.json` 의 depth-1+ 노드 title chain → id 매핑을 런타임에 로드.

**금지**: Confluence 출처를 `search?text=...` 같은 검색 링크로 연결하지 말 것. 반드시 `viewpage.action?pageId=<id>` 직접 링크.

### 배치

- 표·수식 **직후** 한 줄 아래.
- 단락 **끝**에 한 번.
- 답변 맨 마지막에 몰아서 쓰지 않는다 (근거 ↔ 주장 대응이 애매해짐).

## 3) 원문 보존

| 종류 | 규칙 |
|------|------|
| 수치/상수 | 원문 표기 그대로 (예: `999,999`, `MetaSwitchCoolTime = 3`). 단위 변환 금지. |
| 공식 | 한 줄 수식 블록: `` ```수식 ``` `` 또는 Markdown 인용. 기호를 임의로 바꾸지 않는다. |
| 표 | 원본 열 순서·머리글 유지. 필요 시 일부 컬럼만 발췌하되 원본 표기 유지. |
| Mermaid/플로우 | 원본 그대로 코드블록(` ```mermaid `)에 넣는다. 단순화 금지. |
| 이미지 | 가능하면 원문 설명문을 그대로 옮기고, 이미지 자체는 경로 언급만. |
| 영문 식별자 | CamelCase, Enum 값 등은 원문 그대로 (예: `CharacterSkillClass.SkillType = Attack`). |

## 4) 진행 상태 표시 (스트리밍 — **서버만 내보냄, 답변 본문에 쓰지 말 것**)

진행 상태는 **서버 NDJSON 이벤트**(status / thinking / tool_start / tool_end)로만 나간다. UI가 이를 별도 타임라인 블록으로 렌더하고, 완료되면 접어 숨긴다. **에이전트는 답변 본문에 `🔎 탐색 중…`, `📖 정독 중…`, `✅ 답변 준비 완료` 같은 메타 문구를 절대 쓰지 않는다.**

서버(agent-sdk-poc `src/server.py`) 기준 이모지 라벨:

| 단계 | 이모지 | 메시지 형식 |
|------|--------|-------------|
| 분석/계획 | 🧠 | "질문을 분석하고 있습니다..." |
| 내부 사고 | 💭 | `💭 <첫 100자>...` |
| 탐색 (Grep) | 🔎 | `` 🔎 `<pattern>` 검색 중 (<path>) `` |
| 탐색 (Glob) | 📂 | `` 📂 패턴 매칭: `<pattern>` `` |
| 읽기 (Read) | 📖 | `📖 <워크북> / <시트> 읽는 중` (basename이 `content.md`면 의미있는 parent dir 사용) |
| 관계 조회 (KG) | 🔗 | "🔗 시스템 관계 조회 (<tool_suffix>)" |
| 용어 조회 | 🔤 | "🔤 용어 정규화 ..." |
| 종합 | ✨ | "✨ 종합하고 있습니다..." |
| **답변 작성** | ✨ | **"✨ 답변을 작성하고 있습니다..."** — 첫 TextBlock 도착 시, 그리고 툴이 재호출된 뒤 다시 text 라운드가 시작될 때마다 방출. 이후 200자 단위로 `✨ 답변 작성 중... (N,NNN자)` 로 업데이트 (true streaming 시). |
| 완료 | ✅ | 응답 완료 (플라이트가 ResultMessage 를 받으면 자동 표시) |
| 오류 | ❌ | 오류 메시지 |

**UI 요구사항**: 로딩 중 `.progress-head` 는 shimmer/pulse 애니메이션을 유지하여 "살아있음"을 시각적으로 알린다. 텍스트가 정지해도 애니메이션이 돌아가면 사용자는 대기 상태임을 인지한다. 정지 UI 금지.

- 각 이벤트는 한 줄이며 최대 120자.
- 원문 경로/키워드는 백틱(`)으로 감싼다.
- UI는 각 tool_call 을 접힘/펼침 블록으로 표시. 완료 후 전체 타임라인을 자동 접어 답변 본문을 강조한다.

## 5) 모름·불확실

- 문서에서 확인되지 않으면 **"문서에서 확인되지 않습니다."** 명시. 이어서 **어떤 경로를 탐색했는지** 적는다.
- 추측으로 채우지 않는다. 대신 "가능성 있는 인접 시스템: ..." 형태로 다음 탐색 힌트를 제공한다.
- 원본이 TBD/작성 예정인 경우 그대로 전달 (해석·보완 금지).

예:
```
문서에서 확인되지 않습니다. 아래 경로를 탐색했으나 수치가 명시되어 있지 않습니다.
- `PK_변신 및 스킬 시스템/변신.md` — 쿨타임 존재만 언급
- `PK_HUD 시스템/HUD_전투.md` — 별도 상수 테이블 없음

관련 가능성: `ContentSetting.xlsx` 상수 테이블, `PK_밸런싱` 문서.
```

## 6) 금지 사항

1. **답변 본문에 진행 이모지/메타 문구 금지** — 🧠/🔎/📖/✅ 는 SSE status 전용. 답변은 `## 결론` 으로 바로 시작한다. "탐색 중…", "정독 중…", "답변 준비 완료" 같은 메타 서술 절대 금지.
2. **출처 축약 금지** — `(출처: 위와 동일)`, `(상동)`, `(같음)` 등 사용 금지. 매번 전체 경로+섹션을 적는다. UI가 출처 카드로 dedup·그룹화한다.
3. **🚨 내부 파일 시스템 경로 노출 절대 금지** — `packages/xlsx-extractor/output/...` / `packages/confluence-downloader/output/...` / `../xlsx-extractor/...` / `../confluence-downloader/...` 같은 **파일 시스템 경로를 답변 본문 어디에도 쓰지 말 것** (인용·리스트·관련 문서·"더 볼만한 곳"·문단 중간 어디든). 사용자용 라벨(`워크북명.xlsx / 시트명 시트`, `Confluence / 공간 / ... / 페이지`)만 노출. 서버에 sanitize 장치가 있지만 **에이전트가 애초에 쓰지 않는 것**이 원칙.
4. **🚨 내부 인덱스 파일 인용 금지** — `index/MASTER_INDEX.md`, `index/TERM_INDEX.md`, `index/summaries/...` 는 **탐색 보조 도구**다. 답변의 근거·출처로 인용하지 말 것. 실제 원본 `content.md`를 Read 해서 그 워크북/시트·페이지를 인용하라. 카탈로그 질문이라도 MASTER_INDEX 자체를 출처로 쓰지 말고, 실제 시스템 content.md 여러 개를 Read 해서 그것들을 인용.
5. **존재하지 않는 파일 나열 금지** — "더 볼만한 곳"에는 **실제 탐색한 문서** 또는 **문서 내 명시적 참조**만. 불확실한 일반명(`content.md`, `설정.md`)을 경로 없이 나열하지 말 것. 문서 내 참조로만 확인된 경우 "※ 실제 경로 미확인" 표기.
6. **추측 답변** — 문서에 없는 값을 만들어 채우지 않는다.
7. **요약 대체** — 핵심 수치·공식을 "적당히"로 요약하지 않는다. 필요 시 원문 인용.
8. **출처 누락** — 사실 주장은 반드시 인용. 인용 없는 주장은 "내부 지식에 기반한 해석입니다"로 명시.
9. **마크다운 남용** — `**굵게**` 는 핵심 결론/수치에만. 문단 전체 굵게 금지.
10. **장식 이모지** — 🎉, 💯, 🔥 등 사용 금지.
11. **자화자찬 메타** — "포괄적으로", "종합적으로", "탁월하게" 금지.

## 사용법 (에이전트 CLAUDE.md에 포함)

이 스킬은 CLAUDE.md에서 다음과 같이 끌어 쓴다:

```markdown
## 답변 포맷

이 프로젝트의 답변 포맷·인용 규칙·진행 상태 표시는 `qna-output-format` 스킬의 정의를 따른다.
세부 규칙: https://… 또는 프로젝트 내 `.claude/skills/qna-output-format/SKILL.md`.

(핵심만 요약)
- 결론→근거→관련→더 볼만한 곳 4단
- 모든 주장 뒤 `(출처: <경로> § <섹션>)`
- 수치/공식/표는 원문 보존, 모름은 "확인되지 않습니다"
```
