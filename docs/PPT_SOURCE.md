# Project K AI 기획 어시스턴트 — PPT 소스 자료

> 이 문서는 프로젝트 전체를 PPT로 만들기 위한 소스 자료입니다.
> 각 섹션이 PPT의 한 슬라이드(또는 슬라이드 그룹)에 대응됩니다.

---

## 1. 프로젝트 한 줄 요약

**모바일 MMORPG "Project K"의 전체 기획 지식을 AI가 이해하고,
기획자의 작업을 실시간으로 보조하는 AI 에이전트 시스템**

---

## 2. 문제 정의 — 왜 필요한가?

### 현실의 고통

| 문제 | 상세 |
|------|------|
| **기획서 분산** | 104개 엑셀 + 490개 Confluence 페이지 + 11개 PPT에 기획 지식이 흩어져 있음 |
| **전체 파악 불가** | 기획자 한 명이 전체 시스템(100+개)을 파악하는 것은 물리적으로 불가능 |
| **시스템 간 충돌** | "변신"과 "스킬"이 같은 수치를 다르게 정의해도 아무도 모름 |
| **암묵지 유실** | 핵심 기획자 퇴사 시 "왜 이렇게 설계했는지" 영구 유실 |
| **온보딩 비용** | 신규 입사자가 전체 시스템을 파악하는 데 수 주 소요 |

### AI가 해결하면?

- 어떤 시스템에 대해서든 **즉시 정확한 답변** (출처 포함)
- 새 기획서가 기존과 **충돌하는지 자동 감지**
- 기획서 작성 시 **실시간으로 관련 정보 제안**
- 퇴사해도 **지식이 AI에 남아있음**

---

## 3. 최종 비전

> Project K의 기획 내용을 가장 잘 아는 AI 전문가가,
> 기획자가 작업할 때 **실시간으로 옆에서 도와주는 것**.

### 진화 방향

```
[1] QnA 봇         ← 현재 여기! (95% 달성)
 ↓
[2] 실시간 Assist    기획서 작성 중 실시간 피드백 & 가이드
 ↓
[3] 프로액티브 리포트  충돌/공백/갱신 필요 영역 자동 감지
 ↓
[4] 역할별 확장      개발자 구현 가이드 / QA TC 자동 생성 / 사업팀 해설
 ↓
[∞] 게임 개발 전체 생명주기를 관통하는 AI 에이전트 플랫폼
```

**핵심 통찰**: 같은 지식 베이스 위에서 **페르소나(역할)와 트리거(시점)만 달라지는 것**.
지식 베이스가 탄탄할수록 모든 확장이 자연스러워진다.

## 5. 1단계: 지식화 — 기획서를 AI가 읽을 수 있는 형태로

### 변환 대상

| 유형 | 규모 | 변환율 | 비고 |
|------|------|--------|------|
| **Excel 기획서** | 104파일, 635시트 | **98.1%** (623시트) | 핵심 시스템 기획서 |
| **Confluence** | 490페이지 | **99.8%** (489개) | REST API 직접 추출 |
| **PPTX** | 11개 | 진행 중 | 후순위 |

### Vision-First 변환 파이프라인

기존 방식의 한계: Excel 정보의 80%가 **셀 밖**(도형, 플로우차트, 이미지)에 존재
→ openpyxl로 셀만 읽으면 20%만 추출됨

**전략 전환 (Vision-First)**:

```
[Step 1] Excel 시트 → 스크린샷 캡처 (COM headless)
           ↓
[Step 2] Claude Vision API → 전체 레이아웃/도형/텍스트 AI 인식
           ↓
[Step 3] OOXML 파싱 → 정확한 수치/수식/숨겨진 셀 보강
           ↓
[Step 4] 합성 → 최종 구조화 Markdown (content.md)
```

### 변환 성과

- 시범 변환: **2KB → 200KB** (100배 정보량 확보)
- 도형 추출: 327/327 (100%), 연결선 매핑: 123/123 (100%)
- 전체 변환: 10.3시간, 7.8M 토큰 사용

### Confluence 변환 전략

기존 PDF 경유(Vision API) 대신 **REST API에서 직접 추출** — 비용 0, 속도 수십 배

```
Confluence REST API → Storage Format(HTML/XML)
                    → BeautifulSoup + markdownify
                    → Markdown + 원본 이미지 다운로드
```

결과: 490페이지, 489개 content.md, ~2,195 이미지

## 7. 핵심 기술: Agent 파이프라인

### 왜 단순 RAG로는 부족한가?

| | 단순 RAG | Agent 파이프라인 |
|---|---|---|
| 접근 | 키워드 매칭 → 검색 → 답변 | **질문 이해 → 전략 수립 → 검색 → 답변 → 자체 검증** |
| 정확도 | 47% (기준선) | **95%** |
| 약점 | 잘못된 문서 검색, 보수적 답변 | Planning이 올바른 문서를 사전에 선택 |

### Agent 4원칙 아키텍처

```
질문 입력
    ↓
┌─────────────────────────────────────────┐
│  [1] Planning (Claude Sonnet)           │
│  • 질문 분석 + 의도 파악                  │
│  • 167개 기획서 중 관련 문서 선택          │
│  • Knowledge Graph로 연관 시스템 탐색      │
│  • 검색 전략 수립 (어디서, 무엇을 찾을지)   │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│  [2] Search (하이브리드 4레이어)          │
│  • Layer 1: 구조적 검색 (섹션명 키워드)    │
│  • Layer 2: KG 확장 (관련 시스템 BFS)     │
│  • Layer 3: 벡터 검색 (임베딩 유사도)      │
│  • Layer 4: 풀텍스트 (정확 문자열 매칭)    │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│  [3] Answer (Claude Sonnet, temp=0)     │
│  • 수집된 증거 기반 답변 생성              │
│  • 9개 가드레일 규칙 (할루시네이션 방지)    │
│  • 역할별 맞춤 답변 (기획자/프로그래머/QA)  │
│  • 출처 명시 [워크북/시트]                 │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│  [4] Reflection (Claude Haiku)          │
│  • 답변 품질 자체 검증                    │
│  • 부족하면 → 다른 키워드로 재검색 1회     │
│  • 충분하면 → confidence 판정 후 최종 반환  │
└─────────────────────────────────────────┘
               ↓
          최종 답변 반환
    (답변 + 출처 + confidence + trace)
```

## 9. 핵심 기술: 평가 체계 (LLM-as-Judge)

### 3단계 평가 파이프라인

```
[1단계] 질문 생성
    기획서(content.md) → LLM → 자연어 질문 + 기대 정답 + 핵심 팩트
    69개 질문 (7카테고리: 사실, 연관, 수치, 플로우, UI, 메타, 트랩)

[2단계] 답변 생성
    GT 질문 → Agent 파이프라인 → 시스템 답변

[3단계] Judge 채점
    기대 정답 vs 시스템 답변 → Judge LLM → 8축 점수
```

### 8축 채점 기준

| 축 | 측정 대상 | 가중치 |
|----|----------|--------|
| intent_alignment | 질문 의도에 맞는 답변인가 | 동일 |
| factual_accuracy | 사실이 정확한가 | 동일 |
| completeness | 핵심 정보가 빠짐없는가 | 동일 |
| no_misinterpretation | 오해/왜곡이 없는가 | 동일 |
| source_fidelity | 출처를 정확히 인용했는가 | 동일 |
| actionability | 실무에 바로 활용 가능한가 | 동일 |
| scope_match | 답변 범위가 적절한가 | 동일 |
| freshness | 최신 정보를 반영했는가 | 동일 |

**판정**: PASS(평균>=4.0 AND 최소>=3), PARTIAL(평균>=3.0), FAIL

## 10. 할루시네이션 방지 — 트랩 가드레일

### 문제: AI가 없는 내용을 지어내는 것

기획 QnA에서 가장 위험한 것은 **"그럴듯하지만 틀린 답변"**

### 해결: 9개 가드레일 규칙

핵심 규칙:
- **Rule #7**: "A 시스템에 대한 내용이 있음" ≠ "A 시스템의 B 기능이 있음"
  - 예: 창고 시스템 문서가 있어도 "내구도 회복" 기능이 언급되지 않으면 → "정의되지 않은 기능"
- **Rule #9**: 컨텍스트에 **한 번도 언급되지 않는** 기능/동작 → "기획서에 정의되어 있지 않습니다"
  - 이 규칙은 다른 모든 규칙보다 우선

### 결과

- 트랩 질문 9/9 **100% 방어**
- "없는 기능"을 물어봐도 절대 지어내지 않음

## 12. 확장 비전 — 역할별 AI 에이전트

| **아트** | 리소스 요구사항 추출 | "이 기획에 필요한 아트 에셋 23건" |

### JIRA + Perforce + Slack + Email 통합

```
Slack 논의 → Email 승인 → Confluence 문서화 → 기획서 반영 → JIRA 태스크 → Perforce 커밋
```

어떤 설계 결정이든 **전체 맥락 체인**을 AI가 자동 복원.
현재 이 추적은 사람 기억에 100% 의존 — **퇴사 시 영구 유실**.

### 새로 가능해지는 것

| 능력 | 설명 |
|------|------|
| **의사결정 고고학** | "이 수치가 왜 이렇게 설정됐는지" → Slack 논의 + 승인 이력 자동 복원 |
| **리스크 조기 경보** | JIRA 지연 + Slack 논의 급증 → 설계 문제 가능성 자동 감지 |
| **버스 팩터 경고** | "변신 시스템: 버스 팩터 1 (A님에게 90% 의존). 문서화 긴급 권장" |
| **전파 누락 감지** | Slack에서 결정됐지만 공식 문서에 미반영 → 자동 알림 |
| **밸런스 타임라인** | 게임 데이터 수치 변경 이력 + 각 변경의 "왜" 자동 추적 |
