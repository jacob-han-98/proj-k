# Project K AI 기획 어시스턴트 - 비전 & 로드맵

## 왜 이 프로젝트가 필요한가?

모바일 MMORPG "Project K"의 기획 문서는 104개 엑셀, 296개 PDF, 11개 PPT에 분산되어 있다.
기획자 한 명이 전체 시스템을 파악하기 어렵고, 시스템 간 충돌이나 누락을 사전에 발견하기 힘들다.

AI가 이 모든 기획 지식을 구조화하여 보유하면:
- 어떤 시스템에 대해서든 즉시 답변 가능
- 새 기획서가 기존 시스템과 충돌하는지 자동 감지
- 기획서 작성 시 실시간으로 관련 정보와 의견 제공

## 최종 비전

> Project K의 기획 내용을 가장 잘 아는 AI 전문가가,
> 기획자가 Confluence나 Excel에서 작업할 때 실시간으로 옆에서 도와주는 것.

---

## 단계별 상세 스펙

> **설계 방침**: 2~5단계는 **PoC(Proof of Concept)**에 집중한다.
> UI/UX보다 **Backend API를 탄탄하게** 만들어 핵심 가치를 검증한다.
> 5단계 완료 후 경영진 피칭 → 승인 시 6단계에서 사내 서비스로 전환한다.

### 1단계: 지식화 (Knowledge Extraction)

**목표**: 모든 기획 문서를 AI가 검색/이해 가능한 구조화된 지식으로 변환

**현재 상태**: Excel 변환 완료 (623/635 시트, 98.1%), PDF/PPTX는 확장 트랙으로 분리

#### 1-1. 문서 변환 파이프라인 - Vision-First 방식

**전략 전환 (ADR-005)**:
- 기존: Excel 파싱(openpyxl) 먼저 → Vision API로 보정
- **신규: Vision API 먼저 → Excel 데이터로 보강**

```
[1차] 스크린샷 → Vision API  : 레이아웃, 도형, 플로우차트, 이미지 내 텍스트
[2차] Excel 데이터 파싱 보강  : 정확한 수치, 숨겨진 셀, 수식, 데이터 테이블
[합성] 두 결과 병합           : 최종 구조화된 Markdown
```

- PDF: Vision API로 페이지 분석 (기존 PyMuPDF 텍스트 추출은 보강용)
- PPTX: Vision API로 슬라이드 분석 (기존 python-pptx는 보강용)
- 서버 환경: headless로 스크린샷 캡처 + 변환 자동화

**검증 완료 (기존 방식 기준)**:
- 시범 변환(변신/스킬 시스템)에서 2KB → 200KB (100배) 정보량 확보

#### 1-2. 지식 구조화
- 변환된 Markdown을 시스템별/주제별로 인덱싱
- 시스템 간 관계 그래프 (knowledge_graph.json) 자동 생성/업데이트
- 용어 사전 자동 구축 (게임 내 고유 용어, 약어, 시스템명)
- 청크 분할 전략: RAG에 최적화된 단위로 분리

#### 1-3. 일괄 변환 실행
- 우선순위: 7_System(67개) → Confluence PDF(296개) → 9_MileStone(28개) → PPTX(11개)
- 변환 품질 자동 검증 파이프라인

#### 산출물
- Vision-First 변환 파이프라인 (headless 서버 동작)
- `_knowledge_base/` 전체 채워진 상태
- 시스템 관계 그래프, 용어 사전
- 검색 가능한 인덱스

### 2단계: QnA API (PoC)

**목표**: Project K 기획에 대해 질문하면 정확한 답변을 주는 **Backend API**

**방침**: API 품질에 집중. UI는 검증용 최소 데모만.

#### 2-1. RAG 파이프라인
- 지식 베이스 → 벡터 임베딩 → 벡터 DB
- 질문 → 관련 문서 검색 → Claude API로 답변 생성
- 출처 표시: 어떤 기획서의 어느 부분에서 정보를 가져왔는지

#### 2-2. API 설계
- REST API
- 엔드포인트: `/ask`, `/search`, `/related-systems`
- 대화 히스토리 관리 (멀티턴)

#### 2-3. 검증용 데모 UI
- API 동작 확인용 최소 웹 인터페이스
- 답변 정확도 측정 및 피드백 수집

#### 산출물
- Backend API 서버 + API 문서
- 검증용 데모 UI

### 3단계: 데이터 확장 & 동기화

**목표**: 기획 리뷰(4단계)에 필요한 **완전하고 최신인 데이터셋** 확보

**방침**: QnA PoC(2단계)에서 드러난 커버리지 부족 영역을 우선 보강. 자동 동기화 기반 구축.

**왜 기획 리뷰 전에 필요한가?**
- 기획 리뷰는 "전체 기획서 간 충돌/누락 감지"가 핵심 → **데이터가 불완전하면 오탐/미탐 발생**
- Confluence에 의사결정 히스토리, 정책 문서가 있음 → 이것 없이는 "왜 이렇게 설계했는지" 답변 불가
- Perforce 동기화 없이는 기획서가 이미 수정된 상태인데 구 버전으로 리뷰하는 사태 발생

#### 3-1. Confluence 다운로드 — 완료 (ADR-010)
- **전략 전환**: PDF 변환(Vision-First) 대신 **Confluence REST API에서 직접 추출**
- Storage Format → BeautifulSoup + markdownify → Markdown + 이미지 로컬 저장
- **결과**: 490페이지 다운로드, 489개 content.md, ~2,195 이미지, 8.2GB
- 도구: `packages/confluence-downloader/` (client.py, converter.py, run.py)
- **다음**: 변환 품질 검증 → QnA 인덱싱 통합 (ChromaDB)

#### 3-2. PPTX 변환 완료 (11개)
- 컨셉/방향성 문서 — 기획 의도(Why) 이해에 필수
- 기존 convert_pptx.py 또는 Vision-First 적용

#### 3-3. Perforce 원본 동기화
- Excel 원본을 Perforce에서 자동 가져오기
- 변경 감지 (changelist 모니터링) → 변경된 파일만 자동 재변환
- 기획서 버전 관리: 현재 버전 vs 이전 버전 diff 추적

#### 3-4. Knowledge Graph 확장
- Confluence/PPTX 데이터로 knowledge_graph.json 보강
- 시스템 간 관계뿐 아니라 의사결정 맥락, 설계 이유 연결
- QnA 검색 인덱스(ChromaDB) 자동 갱신

#### 3-5. 데이터 커버리지 맵
- 전체 시스템 목록 대비 기획서 커버리지 자동 측정
- 커버리지 부족 영역 → 기획 리뷰(4단계)에서 "미기획 영역" 리포트의 근거

#### 산출물
- Confluence PDF 변환 결과 (지식 베이스 통합)
- PPTX 변환 완료
- Perforce 자동 동기화 파이프라인
- 확장된 Knowledge Graph + 검색 인덱스
- 데이터 커버리지 리포트

### 4단계: 기획 리뷰 API (PoC)

**목표**: 기존 기획의 문제점 감지 + 신규 기획서 리뷰 **API**

**방침**: 분석 로직과 API에 집중. 리포트는 API 응답으로 제공.

#### 4-1. 기존 기획 분석
- 시스템 간 충돌 감지 (같은 개념을 다르게 정의한 곳)
- 내역 부족 감지 (참조하지만 상세 기획이 없는 시스템)
- 수치 불일치 감지 (같은 테이블/공식의 값이 문서마다 다른 경우)
- 미구현/미완성 기획 식별

#### 4-2. 신규 기획서 리뷰
- 기획서 업로드 → AI가 기존 시스템과의 정합성 검토
- 누락된 고려사항 제안
- 유사 시스템 사례 비교 제시

#### 산출물
- 기획 리뷰 API (기존 분석 + 신규 리뷰)
- 분석 결과 리포트 포맷

### 5단계: 실시간 어시스턴트 API (PoC)

**목표**: 기획자의 작업 맥락에 맞는 실시간 피드백을 제공하는 **Backend API**

**방침**: 실시간 피드백 로직에 집중. Confluence 플러그인 등 UX는 6단계로.

#### 5-1. 맥락 인식 API
- 작성 중인 문서 내용을 입력하면 관련 기존 기획 정보 반환
- 기존 시스템과 충돌 가능성 분석
- 용어 일관성 검사
- 빠진 항목 제안 (체크리스트 기반)

#### 5-2. 스트리밍 응답
- WebSocket 또는 SSE 기반 실시간 응답
- 문서 변경분(diff) 단위 입력 지원

#### 산출물
- 실시간 어시스턴트 API
- 맥락 인식 + 피드백 엔진

### 경영진 피칭 마일스톤 (5단계 → 6단계 게이트)

> 5단계 PoC 완료 시점에 경영진과 주요 이해관계자에게 **프로젝트 가치를 시연**한다.
> 승인을 받아야 6단계(사내 서비스화) 진행.

**시연 시나리오 (예시)**:
1. **QnA 데모**: "스킬 시스템과 변신 시스템이 어떻게 연동되나?" → 67개 기획서를 종합한 즉답 (출처 포함)
2. **리뷰 데모**: 신규 기획서 초안 업로드 → 기존 시스템과의 충돌/누락 자동 감지
3. **실시간 데모**: 기획서 작성 중 관련 정보 실시간 제안

**피칭 포인트**:
- 신규 입사자 온보딩 시간 단축 (수주 → 수일)
- 기획 충돌/누락의 사전 감지 → 개발 재작업 비용 절감
- 팀원 퇴사 시 암묵지 유실 방지
- 향후 게임 데이터 + 서버 코드까지 연결 시 기획↔구현 3중 교차 검증 가능

### 6단계: 사내 서비스 (Production)

**목표**: PoC를 **사내 정식 서비스**로 전환. 서비스 배포 + 접근 권한 관리.

> 3단계에서 구축한 데이터 동기화 파이프라인을 프로덕션 수준으로 강화.

#### 6-1. 데이터 동기화 고도화
- 3단계 Perforce/Confluence 동기화를 프로덕션 안정성으로 강화
- 게임 데이터 Excel + 서버 코드 레포지토리 연동 확대
- 변경 감지 → 자동 재변환 → 지식 베이스 최신 유지 (무중단)

#### 6-2. 서비스 인프라
- 사내 서버 또는 클라우드 배포
- 가용성, 모니터링, 로깅
- API 인증 및 사용량 관리

#### 6-3. 접근 권한 및 배포
- 사내 구성원 계정 연동 (SSO / LDAP)
- 역할별 권한 부여 (기획자, 개발자, QA, PD 등)
- 사용 가이드 및 온보딩 자료

#### 6-4. 서비스 UX
- Confluence 플러그인 또는 브라우저 확장
- 사이드바/팝업 형태의 어시스턴트 UI
- 알림 수준 조절 (적극적 제안 vs 요청 시만)

#### 산출물
- 사내 배포된 정식 서비스
- 프로덕션 수준 데이터 동기화 파이프라인
- 사용자 관리 시스템
- 서비스 UX (Confluence 플러그인 등)

### 7단계: 업무 프로세스 통합 (후보)

> 2026-03-08 논의. 6단계 안정화 이후 검토할 **후보 단계**.
> 기획 지식(1~5단계)에 **업무 프로세스 데이터**를 통합하여,
> "기획이 어떻게 설계되었는가"를 넘어 **"기획이 어떻게 실행되고 있는가"**까지 답하는 AI로 확장.

#### 7-0. 통합 대상 및 우선순위

| 우선순위 | 소스 | 핵심 가치 | 기술 난이도 | 보안 민감도 |
|---------|------|----------|-----------|-----------|
| 1순위 | **JIRA** | 기획→구현 생명주기 추적 | 낮음 (REST API 성숙) | 낮음 |
| 2순위 | **Perforce 로그** | 변경 히스토리, 누가/언제/왜 | 중간 (P4 triggers/journal) | 낮음 |
| 3순위 | **Slack** | 비공식 맥락, 암묵지 | 중간 (Events API + 필터링) | 중간 (DM 제외 필수) |
| 4순위 | **Email** | 공식 승인/결정 기록 | 높음 (Exchange/Gmail API + 파싱) | 높음 (개인정보) |

#### 7-1. JIRA 통합

**연동 범위**:
- 프로젝트 내 모든 이슈 (Epic / Story / Task / Bug / Sub-task)
- 이슈 상태 변화 히스토리 (Created → In Progress → Done → Closed)
- 이슈 간 링크 (blocks, is blocked by, relates to, duplicates)
- 스프린트/마일스톤 배정 정보
- 커스텀 필드 (기획서 링크, 시스템 분류 등)
- 댓글 (구현 논의, QA 재현 조건 등)

**새로 가능해지는 것**:

1. **기획 구현 상태 자동 대시보드**
   - 기획서에 정의된 기능 ↔ JIRA 이슈 자동 매핑
   - 상태 판정: 미착수 / 개발중 / 완료 / QA중 / 라이브
   - "변신 시스템 3단계 기획, 구현 상태가 어때?" → JIRA 기반 즉답
   - PD가 수동 관리하는 진행 현황표를 자동 생성

2. **버그 패턴 분석**
   - 특정 시스템에 버그가 집중되는 패턴 자동 감지
   - 반복 발생 버그 유형 클러스터링
   - "스킬 시스템 관련 버그 중 아직 열려있는 것 몇 개야?" → 즉답 + 심각도별 분류

3. **기획 변경 → 영향 JIRA 자동 태깅**
   - 기획서가 수정되면 관련 진행중 JIRA 이슈를 자동 식별
   - "이 기획 변경으로 인해 PK-1234, PK-1567 태스크가 영향받을 수 있습니다"

4. **스프린트 리스크 사전 감지**
   - 스프린트 내 태스크의 관련 기획서 복잡도 분석
   - 기획서 간 의존성이 높은 태스크가 같은 스프린트에 몰려있으면 경고

#### 7-2. Perforce 로그 통합

**연동 범위**:
- Changelist 메타데이터 (번호, 설명, 작성자, 일시, 영향 파일 목록)
- 게임 데이터 Excel 파일의 변경 diff (수치 변경 추적)
- 서버 코드 변경과 JIRA 이슈 연결 (커밋 메시지 내 이슈 키 파싱)
- 브랜치/스트림 정보

**새로 가능해지는 것**:

1. **밸런스 변경 타임라인 자동 생성**
   - 게임 데이터 파일의 수치 변경 이력을 자동 추적
   - 각 변경에 JIRA 이슈 + Slack 논의를 연결하여 "왜 바꿨는지" 맥락 포함
   - "공격력 계수가 현재 1.1인데, 변경 히스토리 알려줘"
     → v1.0 (1/10): 1.0→1.2 (PK-345: 초반 밸런스 상향)
     → v1.3 (2/15): 1.2→0.9 (PK-567: PvP 밸런스 긴급 패치)
     → v2.0 (3/01): 0.9→1.1 (PK-789: 중간값 조정)

2. **기획↔구현 시간차 분석**
   - 기획서 확정일 → Perforce 첫 커밋일 → JIRA 완료일 사이의 리드타임 측정
   - "평균적으로 기획 확정 후 구현까지 몇 일 걸려?" → 시스템별 통계

3. **무단 변경 감지**
   - 기획서 변경 없이 게임 데이터가 수정된 경우 자동 플래그
   - "기획서에 없는 데이터 변경이 3건 감지되었습니다" → 하드코딩 or 누락 기획

#### 7-3. Slack 통합

**연동 범위** (보안 원칙: **공개 채널만**, DM 제외):
- 기획/개발/QA 관련 공개 채널 메시지
- 스레드 전체 (논의 맥락 보존)
- 파일 첨부 (스크린샷, 문서 공유)
- 리액션 (의사결정 투표 등)
- 채널 목록 및 참여자 정보

**노이즈 필터링 전략**:
- 대상 채널 화이트리스트 (예: #기획-변신, #개발-스킬, #qa-버그리포트)
- 키워드/시스템명 기반 관련성 필터링
- 잡담/인사 등 비업무 메시지 자동 제외 (LLM 분류 또는 규칙 기반)
- 임베딩 기반 관련성 스코어링 → 임계값 이상만 인덱싱

**새로 가능해지는 것**:

1. **암묵지 발굴 및 문서화 권고**
   - Slack에만 존재하고 기획서/Confluence에 없는 설계 결정 자동 탐지
   - "변신 합성 시스템에 대해 Slack에서 논의된 내용 중 기획서에 반영되지 않은 사항이 7건 있습니다"
   - 문서화 우선순위 제안 (빈도, 관련 시스템 중요도 기반)

2. **"누가 전문가인가" 자동 파악**
   - 시스템별로 Slack에서 가장 많이 답변/논의하는 사람 분석
   - JIRA 처리량 + Perforce 커밋 빈도와 교차
   - "스킬 시스템에 대해 누구한테 물어봐야 해?"
     → 기획: C님 (Slack 답변 62회, 관련 기획서 작성자)
     → 개발: D님 (Perforce 89 CL, JIRA 31건 완료)

3. **지식 사일로 감지 & 버스 팩터 경고**
   - 특정 시스템에 대한 Slack 답변이 한 사람에게 집중되는 패턴 감지
   - 해당 인원 부재 시 리스크 레벨 산출
   - "변신 합성 시스템: 버스 팩터 1 (A님에게 90% 의존). 문서화 긴급 권장."

4. **논의→결정 추적 체인 완성**
   - Slack 논의 → (Email 승인) → Confluence 문서화 → JIRA 태스크 → Perforce 커밋
   - 의사결정의 전체 라이프사이클을 자동 추적
   - "스킬 쿨타임이 왜 5초야?" → Slack 논의(3/15) → PD 승인(3/17) → JIRA PK-1234(3/18) → CL#5678(3/20)

#### 7-4. Email 통합 (선택적)

**연동 범위** (보안 원칙: **업무 승인 메일만 선별**, 개인 메일 제외):
- 기획 승인/반려 메일 (PD, 리드 발신)
- 마일스톤 공지 메일
- 외부 이해관계자 커뮤니케이션 (퍼블리셔 요구사항 등)
- 명시적 동의를 받은 메일만 수집

**새로 가능해지는 것**:

1. **승인 체인 완성**
   - "이 기획 누가 승인했어?" → 승인 메일 원문 + 일시 + 조건부 승인 내용
   - 승인 없이 진행된 기획 자동 플래그

2. **외부 요구사항 추적**
   - 퍼블리셔/파트너 요청사항 → 기획 반영 여부 자동 추적
   - "퍼블리셔가 요청한 사항 중 미반영된 것 있어?" → 즉답

#### 7-5. 교차 시너지 — 7단계 통합 시 새로 가능해지는 것

단일 소스로는 불가능하고, **교차해야만 보이는** 인사이트:

1. **의사결정 고고학 (Decision Archaeology)**
   ```
   Slack 논의 → Email 승인 → Confluence 문서화 → 기획서 반영 → JIRA 태스크 → Perforce 커밋
   ```
   어떤 설계 결정이든 **전체 맥락 체인**을 자동 복원.
   현재 이 추적은 사람 기억에 100% 의존 — 퇴사 시 영구 유실.

2. **프로젝트 리스크 조기 경보 시스템**
   - JIRA 지연 + Slack 해당 시스템 논의 급증 → **설계 문제 가능성**
   - 기획서 대폭 수정 + 관련 JIRA 없음 → **기획 변경이 개발에 미전달**
   - Perforce 커밋 급감 + JIRA 상태 정체 → **블로커 발생 의심**
   - 특정 시스템 버그 JIRA 집중 + 기획서 최근 미수정 → **설계 재검토 필요**

3. **기획 완성도 종합 스코어카드**
   | 시스템 | 기획서 | Confluence | JIRA | 코드 | 데이터 | Slack 논의 | 종합 |
   |--------|--------|-----------|------|------|--------|-----------|------|
   | 변신 | 상세 | 논의 충분 | 완료 | 구현됨 | 반영됨 | 활발 | A |
   | 스킬연계 | 초안 | 논의 없음 | 미생성 | 없음 | 없음 | 언급 3회 | D |

4. **회의 효율화 제안**
   - 회의 전: "이 안건은 Slack #기획-스킬 (3/5~3/7)에서 이미 결론 도달. 회의 불필요할 수 있음"
   - 회의 후: 회의 결과가 JIRA/기획서에 반영됐는지 자동 추적

5. **결정 전파 누락 감지 (Decision Propagation Gap)**
   - 조직에서 반복되는 고질적 문제: **Slack에서 결정됐지만 공식 문서에 반영 안 됨**
   - AI가 채널 간 정보 흐름을 교차 대조하여 **전파 누락을 자동 탐지**:
     - Slack에서 결론 도달 + Confluence/기획서 미갱신 → "이 결정이 공식 문서에 반영되지 않았습니다"
     - Slack에서 결정 + Email 공지 미발송 → "관련 이해관계자에게 아직 전파되지 않았습니다"
     - 기획서 변경 + 관련 팀 채널 미공유 → "개발팀/QA팀에 변경 사항이 공유되지 않았습니다"
     - JIRA 상태 변경 + Slack 알림 없음 → "담당자 변경/완료가 관련자에게 미통보"
   - **사전 전파 누락 감지 (Early Signal Gap)** — 확정 전이라도 후속 조직에 미리 알려야 하는 경우:
     - 기획팀에서 신규 시스템/대규모 변경을 검토 중 + 후속 조직(서버, 클라, 아트, QA 등) 채널에 언급 없음
       → "기획팀에서 [X 시스템 변경]을 논의 중입니다. 서버팀/아트팀에 사전 공유가 필요할 수 있습니다"
     - 특히 **여러 조직의 요청을 동시 처리하며 스케줄링 난이도가 높은 후속 조직**(서버팀, 아트팀 등)에게는
       확정 전이라도 가능성을 미리 전파해야 feasibility 검토와 일정 확보가 가능
     - AI가 감지하는 신호:
       - Slack 기획 채널에서 특정 시스템 논의 활성화 (메시지 빈도 급증, 참여자 증가)
       - 기획서 초안/WIP 문서 생성 감지
       - JIRA Epic/Story 생성되었으나 후속 조직 담당자 미배정
     - 알림 예시: "기획팀이 [변신 3단계]를 검토 중입니다 (Slack 논의 3일차, 참여자 5명).
       서버팀 사전 feasibility 검토가 필요할 수 있습니다. 서버팀 예상 영향 범위: [스킬 시스템, 전투 로직]"
   - 결과: **"알고 있는 사람"과 "알아야 하는 사람" 사이의 간극**을 AI가 자동으로 메워줌.
     확정된 결정뿐 아니라, **검토 단계의 가능성까지 선제적으로 전파**하여 후속 조직의 대응 시간을 확보

#### 7-5-1. 개발 조직을 넘어선 확장 가능성

> 결정 전파 누락, 암묵지 유실, 지식 사일로 문제는 **개발팀만의 문제가 아니다.**
> 동일한 아키텍처를 다른 조직에도 적용할 수 있다.

| 조직 | 적용 시나리오 | 기대 효과 |
|------|-------------|----------|
| **사업/퍼블리싱** | 퍼블리셔 요청사항 → 기획 반영 → 개발 구현 추적 | 외부 요구사항 누락 방지, 계약 이행 추적 |
| **QA** | 테스트 케이스 ↔ 기획서 ↔ 버그 리포트 교차 분석 | 테스트 커버리지 가시화, 회귀 테스트 자동 우선순위 |
| **아트** | 아트 리소스 요청(기획) → 제작(JIRA) → 납품(Perforce) 추적 | 아트 에셋 누락/지연 조기 감지 |
| **운영** | 라이브 이슈 → 기획 의도 → 핫픽스 이력 연결 | CS 대응 시 맥락 즉시 파악, 반복 이슈 패턴 분석 |
| **경영/PM** | 전체 프로젝트 건강도 대시보드 | 의사결정 병목, 조직 간 전파 지연, 리스크 한눈에 파악 |

**핵심 인사이트**: 이 시스템은 본질적으로 **"조직의 정보 흐름을 가시화하고 단절을 감지하는 AI"**다.
게임 기획 도메인에서 시작하지만, 검증 후 **회사 전체의 업무 프로세스 인텔리전스**로 확장 가능하다.
이 확장성은 경영진 피칭 시 **장기 비전**으로 제시할 수 있는 강력한 포인트다.

#### 7-6. 기술 고려사항

| 항목 | 상세 |
|------|------|
| **JIRA 연동** | Atlassian REST API v3, Webhook으로 실시간 이슈 변경 수신, JQL 기반 배치 동기화 |

### 8단계: 조직 인텔리전스 (후보)

> 7단계에서 개발팀 중심으로 검증된 업무 프로세스 통합을,
> **회사 전체 공식 Slack + 업무 시스템**으로 확장하여
> 조직 단위의 R&R, 커뮤니케이션 흐름, 의사결정 병목을 분석하는 단계.

#### 8-1. 전사 Slack 모니터링 기반 업무 흐름 분석

**연동 범위**: 회사 전체 공식 채널 (DM 제외, 개인정보 보호 원칙 동일)

**R&R 실태 분석**:
- **공식 R&R vs 실제 업무 담당자 괴리 감지**
  - 조직도상 A팀 소관이지만 실제로는 B님이 항상 답변/처리하는 패턴
  - "변신 시스템 관련 질문에 기획2팀이 담당이지만, 실제 답변의 78%는 기획1팀 A님입니다"
- **회색지대 업무 감지**
  - 어떤 채널에서든 반복적으로 질문이 올라오지만 담당자가 불명확한 영역
  - "이 업무는 누구 소관인지 불분명합니다 (3개 채널에서 7회 질문, 매번 다른 사람이 답변)"
- **병목 인력 식별**
  - 특정 인원에게 과도하게 집중되는 멘션/질문/요청 패턴
  - "A님이 5개 채널에서 일일 평균 23건의 질문에 답변하고 있습니다. 업무 분산이 필요할 수 있습니다"

#### 8-2. 조직 간 커뮤니케이션 흐름 분석

**정보 전달 지연 감지**:
- 조직 A에서 결정/변경 발생 → 조직 B 채널에 전달되기까지의 리드타임 측정
- "기획팀→서버팀 평균 전달 지연: 2.3일, 기획팀→아트팀: 4.1일"
- 특정 사안별 지연 알림: "이 기획 변경이 서버팀에 3일째 미전달입니다"

**우회 소통 패턴 감지**:
- 공식 프로세스를 거치지 않는 직접 요청 패턴
- "원래 기획팀→PM→아트팀 경로인데, 기획팀이 직접 아트팀에 요청하는 경우가 월 15건"
- 우회가 반복되면 공식 프로세스 자체의 재검토가 필요하다는 시그널

**정보 사일로 감지**:
- 동일 주제가 여러 채널에서 별도로 논의되는 패턴
- "변신 시스템 밸런스에 대해 #기획-변신, #개발-전투, #qa-밸런스 3개 채널에서 각각 논의 중. 결론이 다릅니다"

#### 8-3. 의사결정 병목 분석

**의사결정 속도 측정**:
- 안건 제기 → 논의 → 결정 → 전파 → 실행의 각 구간별 소요 시간
- 조직별, 안건 유형별 평균 의사결정 리드타임 통계
- "기획 변경 의사결정 평균 5.2일, 그 중 승인 대기가 3.1일 (60%)"

**병목 구간 자동 식별**:
- 의사결정 체인에서 가장 오래 걸리는 구간과 담당자/조직
- 반복적으로 병목이 되는 역할/프로세스 패턴
- "지난 분기 의사결정 지연 23건 중 18건이 [X 승인 단계]에서 정체"

#### 8-4. 경영 대시보드

| 지표 | 측정 방법 | 활용 |
|------|----------|------|
| **조직 간 전달 지연** | 채널 간 동일 주제 언급 시차 | 프로세스 개선 포인트 발견 |
| **R&R 괴리도** | 공식 담당 vs 실제 답변자 불일치율 | 조직 개편/R&R 재정의 근거 |
| **병목 인력 집중도** | 인당 멘션/질문 응답 빈도 | 인력 충원/업무 재분배 근거 |
| **의사결정 리드타임** | 안건별 제기→결정→실행 소요일 | 프로세스 간소화 대상 식별 |
| **정보 사일로 수** | 동일 주제 분산 논의 채널 수 | 채널 통합/크로스팀 미팅 필요성 |
| **결정 전파율** | 결정 후 관련 채널 공유 비율 | 전파 누락 경향이 높은 조직 코칭 |

#### 8-5. 보안 및 거버넌스

> 전사 Slack 분석은 민감도가 높다. 7단계보다 엄격한 거버넌스가 필요.

- **경영진/CISO 사전 승인** 필수
- **분석 범위 투명 공개**: 모니터링 대상 채널 목록을 전사 공지
- **개인 평가 목적 사용 금지**: R&R 분석은 조직/프로세스 개선 목적에 한정, 개인 성과 측정에 활용하지 않음
- **익명화 옵션**: 경영 대시보드에서 개인명 대신 역할/조직 단위로 집계
- **열람 권한 제한**: 분석 결과는 경영진/PM 등 제한된 역할만 접근
- **옵트아웃 정책**: 특정 채널의 분석 제외 요청 처리 프로세스

#### 산출물
- 전사 커뮤니케이션 흐름 분석 엔진
- R&R 실태 vs 공식 R&R 괴리 리포트
- 조직 간 전달 지연 / 병목 분석 대시보드
- 의사결정 리드타임 통계 시스템
- 거버넌스 정책 및 권한 관리 체계

#### 성공 기준
- 조직 간 정보 전달 지연 평균 30%+ 단축
- R&R 회색지대 식별 → 공식 담당 배정율 80%+
- 경영진이 월례 리뷰에서 대시보드를 실제 활용

## 데이터 적재 로드맵 (후보)

> 2026-03-08 논의. 확정이 아닌 **후보 계획**으로, 실제 진행 시 재검토 필요.

### 적재 순서 및 기대 효과

```
Phase 1  Excel 기획서 (104개)     ← 현재 진행중
Phase 2  PPT 비전문서 (11개)
Phase 3  Confluence PDF (296개)
Phase 4  게임 데이터 Excel
Phase 5  서버 코드
```

| Phase | 데이터 | 새로 가능해지는 것 | 주요 수혜자 |
|-------|--------|-------------------|------------|
| 1 | Excel 기획서 | 시스템 QnA, 시스템 간 충돌/누락 감지, 용어 사전 | 시스템 기획자, 신규 입사자, PD |
| 2 | PPT 비전문서 | 기획 의도(Why) 답변, 비전 정합성 체크 | 기획자 전체, PD |
| 3 | Confluence PDF | 의사결정 추적, 암묵지 보존, 기획 커버리지 맵 | 운영팀, 퇴사 리스크 대비 |
| 4 | 게임 데이터 | 기획↔데이터 정합성 검증, 밸런스 이상치 탐지, 영향도 시뮬레이션 | 밸런스 기획자, QA |
| 5 | 서버 코드 | 기획↔구현 차이 리포트, 유령 기능 발견, 버그 3중 교차 분석, 기술 부채 가시화 | QA, 개발팀, PD |

### 교차 검증 시너지

- Phase 1~3: 설계 의도 + 논의 맥락 → **"왜 이렇게 설계했는가"**
- Phase 1~4: + 실제 데이터 → **"설계대로 데이터가 들어가 있는가"** (실무 가치 급상승)
- Phase 1~5: + 서버 코드 → **"실제로 이렇게 돌아가고 있는가"** (풀스택 기획 AI)

### 킬러 피처 후보

- **기획↔코드 양방향 추적**: 기획서 변경 시 영향받는 코드, 코드 변경 시 관련 기획서 자동 연결
- **기획 미반영 현황 대시보드**: 기획서에 정의됐지만 데이터/코드에 없는 항목 자동 목록화
- **라이브 이슈 진단**: QA 보고 → 기획 의도 + 데이터 값 + 코드 로직 3중 조회
