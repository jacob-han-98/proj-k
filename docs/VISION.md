# Project K AI 기획 어시스턴트 - 비전 & 로드맵

## 왜 이 프로젝트가 필요한가?

모바일 MMORPG "Project K"의 기획 문서는 104개 엑셀, 296개 PDF, 11개 PPT에 분산되어 있다.
기획자 한 명이 전체 시스템을 파악하기 어렵고, 시스템 간 충돌이나 누락을 사전에 발견하기 힘들다.

AI가 이 모든 기획 지식을 구조화하여 보유하면:
- 어떤 시스템에 대해서든 즉시 답변 가능
- 새 기획서가 기존 시스템과 충돌하는지 자동 감지
- 기획서 작성 시 실시간으로 관련 정보와 의견 제공

## 최종 비전

> Project K의 기획 내용을 가장 잘 아는 AI 전문가가,
> 기획자가 Confluence나 Excel에서 작업할 때 실시간으로 옆에서 도와주는 것.

## 단계별 상세 스펙

### 1단계: 지식화 (Knowledge Extraction)

**목표**: 모든 기획 문서를 AI가 검색/이해 가능한 구조화된 지식으로 변환

**현재 상태**: 진행중 (2/104 XLSX, 0/296 PDF, 1/11 PPTX 완료)

#### 1-1. 문서 변환 파이프라인 - Vision-First 방식

**전략 전환 (ADR-005)**:
- 기존: Excel 파싱(openpyxl) 먼저 → Vision API로 보정
- **신규: Vision API 먼저 → Excel 데이터로 보강**

```
[1차] 스크린샷 → Vision API  : 레이아웃, 도형, 플로우차트, 이미지 내 텍스트
[2차] Excel 데이터 파싱 보강  : 정확한 수치, 숨겨진 셀, 수식, 데이터 테이블
[합성] 두 결과 병합           : 최종 구조화된 Markdown
```

- PDF: Vision API로 페이지 분석 (기존 PyMuPDF 텍스트 추출은 보강용)
- PPTX: Vision API로 슬라이드 분석 (기존 python-pptx는 보강용)
- 서버 환경: headless로 스크린샷 캡처 + 변환 자동화

**검증 완료 (기존 방식 기준)**:
- 시범 변환(변신/스킬 시스템)에서 2KB → 200KB (100배) 정보량 확보

#### 1-2. 원본 소스 동기화
- 현재: 임시 복사본 (로컬 폴더, 검증용)
- 향후: Perforce(Excel 원본) + Confluence(PDF) 실시간 동기화
- 변경 감지 → 자동 재변환 → 지식 베이스 최신 유지

#### 1-3. 지식 구조화
- 변환된 Markdown을 시스템별/주제별로 인덱싱
- 시스템 간 관계 그래프 (knowledge_graph.json) 자동 생성/업데이트
- 용어 사전 자동 구축 (게임 내 고유 용어, 약어, 시스템명)
- 청크 분할 전략: RAG에 최적화된 단위로 분리

#### 1-4. 일괄 변환 실행
- 우선순위: 7_System(67개) → Confluence PDF(296개) → 9_MileStone(28개) → PPTX(11개)
- 변환 품질 자동 검증 파이프라인

#### 산출물
- Vision-First 변환 파이프라인 (headless 서버 동작)
- `_knowledge_base/` 전체 채워진 상태
- 시스템 관계 그래프, 용어 사전
- 검색 가능한 인덱스
- Perforce/Confluence 동기화 설정

### 2단계: QnA API (Backend Service)

**목표**: Project K 기획에 대해 질문하면 정확한 답변을 주는 API 서비스

#### 2-1. RAG 파이프라인
- 지식 베이스 → 벡터 임베딩 → 벡터 DB
- 질문 → 관련 문서 검색 → Claude API로 답변 생성
- 출처 표시: 어떤 기획서의 어느 부분에서 정보를 가져왔는지

#### 2-2. API 설계
- REST API 또는 WebSocket
- 엔드포인트: `/ask`, `/search`, `/related-systems`
- 대화 히스토리 관리 (멀티턴)

#### 2-3. 기본 UI
- 간단한 웹 챗 인터페이스
- 답변에 출처 문서 링크 포함

#### 산출물
- Backend API 서버
- 기본 QnA 웹 UI
- API 문서

---

### 3단계: 기획 리뷰 (Design Review)

**목표**: 기존 기획의 문제점 감지 + 신규 기획서 리뷰

#### 3-1. 기존 기획 분석
- 시스템 간 충돌 감지 (같은 개념을 다르게 정의한 곳)
- 내역 부족 감지 (참조하지만 상세 기획이 없는 시스템)
- 수치 불일치 감지 (같은 테이블/공식의 값이 문서마다 다른 경우)
- 미구현/미완성 기획 식별

#### 3-2. 신규 기획서 리뷰
- 기획서 업로드 → AI가 기존 시스템과의 정합성 검토
- 누락된 고려사항 제안
- 유사 시스템 사례 비교 제시

#### 산출물
- 기존 기획 분석 리포트
- 신규 기획서 리뷰 API + UI

---

### 4단계: 실시간 어시스턴트 (Live Assistant)

**목표**: 기획자가 문서를 작성하는 중에 실시간으로 의견을 제공하는 UX

#### 4-1. 문서 모니터링
- Confluence 페이지 변경 감지 (Confluence API webhook/polling)
- Excel 파일 변경 감지 (파일 시스템 watch 또는 SharePoint API)
- 변경된 내용을 실시간 파싱

#### 4-2. 실시간 피드백
- 작성 중인 내용에 대해 관련 기존 기획 정보 제시
- 기존 시스템과 충돌 가능성 경고
- 용어 일관성 검사
- 빠진 항목 제안 (체크리스트 기반)

#### 4-3. UX
- 사이드바 또는 팝업 형태의 어시스턴트 UI
- Confluence 플러그인 또는 브라우저 확장
- 알림 수준 조절 (적극적 제안 vs 요청 시만)

#### 산출물
- 실시간 모니터링 서비스
- Confluence/브라우저 확장 UI
- 피드백 규칙 설정 시스템

## 기술 스택 (잠정)

| 영역 | 후보 기술 | 비고 |
|------|----------|------|
| LLM | Claude API (Anthropic) | Opus/Sonnet 선택적 사용 |
| 벡터 DB | Chroma / Pinecone / pgvector | 로컬 vs 클라우드 결정 필요 |
| Backend | Python (FastAPI) | 기존 변환 스크립트와 통합 |
| Frontend | React / Next.js | QnA UI, 리뷰 UI |
| 임베딩 | Voyage AI / OpenAI Embeddings | Claude 생태계 우선 검토 |
| 문서 감시 | Confluence REST API, Perforce triggers, watchdog | 1단계(동기화) + 4단계 |
| 스크린샷 | headless Excel/PDF 렌더링 (LibreOffice headless 등) | 1단계 |
| 인프라 | AWS (기존 Bedrock 사용중) | |

> 기술 스택 확정은 2단계 착수 시 결정. 상세는 `docs/ARCHITECTURE.md` 참조.

## 성공 기준

| 단계 | 성공 기준 |
|------|----------|
| 1단계 | 전체 문서 90%+ 변환 완료, 지식 그래프 구축 |
| 2단계 | 기획 QnA에 대해 80%+ 정확한 답변 (출처 포함) |
| 3단계 | 기존 기획에서 실제 충돌/누락 5건+ 발견 |
| 4단계 | 기획자 3명+ 실제 업무에서 주 3회+ 사용 |
