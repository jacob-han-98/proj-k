# LLM 기반 GT 질문 생성 & 평가 방법론

## 1. 배경

규칙 기반 질문 생성기(`generate_gt_questions.py`)는 content.md를 파싱하여
"PK_변신 시스템에서 A의 B은?" 같은 기계적 패턴만 생성한다.
실제 기획자/프로그래머/QA/PM이 슬랙에서 물어볼 법한 자연스러운 질문으로 교체하여
QnA 시스템의 실제 성능을 정확히 측정한다.

- 기존: 규칙 기반 파싱 → 기계적 질문 → 키워드 매칭 평가 (97.2%)
- 신규: Claude Sonnet API → 자연어 질문 + 정답 생성 → LLM-as-Judge 의미론적 평가

## 2. 질문 생성 (generate_gt_llm.py)

### 2.1 소스 샘플링

- content.md ~250개 층화 샘플링 (Excel 150 + Confluence 100)
- 문서 크기/복잡도 기준 균등 분배

### 2.2 질문 차원

| 차원 | 내용 |
|------|------|
| **역할** | 기획자, 프로그래머, QA, PM, 아트/연출 |
| **범위** | 단일 문서(Easy), 문서 내 종합(Medium), 크로스 시스템(Hard) |
| **난이도** | Easy 30%, Medium 40%, Hard 20%, 트랩 10% |
| **카테고리** | A(사실), B(시스템간), C(밸런스), D(플로우), E(UI), F(메타), G(Confluence), H(트랩) |
| **구어체** | 슬랙 업무체 + 게임 은어 (쿨감, 딜, 파밍 등) |
| **특수** | 이미지 의존, 모호한 용어, 버전 특정 질문 |

### 2.3 생성 흐름

```
1. content.md ~250개 층화 샘플링
2. 각 파일 → Claude Sonnet → 2~5개 자연어 질문 + 정답 + key_facts 생성
3. KG 기반 문서 세트(2~4개 클러스터) → 세트 통째로 Claude에 전달 → 크로스 시스템 질문
4. 할루시네이션 트랩 45~50개
5. 중복 제거 + 분포 조정 → 최종 ~500개
```

### 2.4 문서 세트 기반 크로스 시스템 질문 (Hard)

실제 업무에서 어려운 질문은 여러 기획서를 종합해야 답할 수 있다.

**세트 구성 방법:**
1. **KG 관계 기반** (주): `knowledge_graph.json`의 `related_systems`에서 연결된 2~4개 시스템 클러스터
2. **벡터 유사도 기반** (보조): ChromaDB에서 의미적으로 유사한 문서 탐색

### 2.5 출력 스키마

```json
{
  "id": "GT-LLM-A-001",
  "query": "변신 합성하면 재료 변신 돌려받을 수 있어?",
  "category": "A",
  "role": "기획자",
  "difficulty": "easy",
  "expected_workbooks": ["PK_변신 및 스킬 시스템"],
  "ground_truth_source": "path/to/content.md",
  "expected_answer": "합성 시 사용한 재료 변신은 모두 소멸한다.",
  "key_facts": ["합성 시 재료 소멸", "되돌리기 불가"],
  "rationale": "기획자가 합성 BM의 리스크를 확인하려는 상황",
  "is_hallucination_trap": false
}
```

**핵심: 질문 생성 시점에 `expected_answer`와 `key_facts`를 함께 생성한다.**
정확한 정답이 있어야 평가가 일관되고 객관적이다.

## 3. 평가 (verify_gt_llm.py)

### 3.1 Stage 1 — 규칙 기반 빠른 필터 (무료)

- 워크북 매칭: 검색 결과에 `expected_workbooks` 포함 여부
- key_facts 핵심 단어 포함 여부
- 용도: 개발 중 빠른 sanity check

### 3.2 Stage 2 — LLM Judge 의미론적 평가 (유료)

Judge에게 전달하는 것:
- 질문 + 기대 정답(`expected_answer`) + 핵심 사실(`key_facts`) + 검색 결과 5개 청크

**기본 8축 (각 1~5점, 총 40점 만점)**

| # | 축 | 평가 질문 |
|---|---|---|
| 1 | **의도 부합** | 질문자의 의도를 정확히 파악한 답변이 가능한가? |
| 2 | **사실 정확성** | 검색 결과의 사실관계가 기대 정답과 일치하는가? |
| 3 | **설명 충분성** | 기획 전문가가 납득할 충분한 설명이 가능한가? |
| 4 | **오해 방지** | 검색 결과가 오해를 유발할 여지가 없는가? |
| 5 | **원문 충실도** | 검색 결과가 원본 문서 내용을 정확히 반영하는가? |
| 6 | **실행 가능성** | 역할에 맞는 실행(구현/테스트/의사결정)이 가능한가? |
| 7 | **범위 적절성** | 질문 스코프에 맞는 답변 범위인가? |
| 8 | **최신성** | 검색 결과가 최신 버전의 정보인가? |

**보너스 1축 (+0~2점)**

| # | 축 | 조건 |
|---|---|---|
| 9 | **암묵적 전제 포함** | 답변에 필요한 전제 조건/선행 지식이 함께 검색됐는가? |

### 3.3 판정 기준

- **PASS**: 기본 8축 평균 ≥ 4.0 AND 모든 축 ≥ 3
- **PARTIAL**: 기본 8축 평균 ≥ 3.0 AND 모든 축 ≥ 2
- **FAIL**: 나머지
- 보너스는 판정 등급에 영향 없음 (품질 지표로 활용)

## 4. 비용

| 항목 | 비용 |
|------|------|
| 질문 생성 (~250 API 호출) | ~$5 |
| LLM Judge 평가 (500 질문) | ~$7 |
| **합계** | **~$12** |

## 5. 실행 순서

```
Phase 1: 1파일 테스트 → 질문 품질 확인           ($0.01)
Phase 2: 10파일 → 30질문 + Stage 1 평가          ($0.10)
Phase 3: 30질문 LLM Judge 테스트                  ($0.50)
Phase 4: 크로스 시스템 30클러스터 → 50질문         ($0.30)
Phase 5: 전체 250파일 → 500질문 + 2단계 평가      ($12)
```

## 6. CLI

```bash
# 질문 생성
python -m eval.generate_gt_llm --sample 5         # 5파일 테스트
python -m eval.generate_gt_llm --dry-run           # 샘플링 계획만
python -m eval.generate_gt_llm                    # 전체 실행

# 평가
python -m eval.verify_gt_llm --stage1-only         # 무료 평가만
python -m eval.verify_gt_llm --sample 20           # 20개만
python -m eval.verify_gt_llm                       # 전체 (Stage 1+2)
```
