## 2. 평가 방법론

### 2.1 평가 설계

**질문 세트 구축 방법**:
1. 실제 629개 content.md를 카테고리별로 스캔
2. 각 카테고리에서 실제 답변 가능한 질문 추출
3. 4개 직군별로 관점을 다르게 설계 (동일 시스템도 직군별 다른 질문)
4. `eval/questions.json`에 질문 + 기대 키워드 + 출처 기록

**왜 키워드 기반 평가인가**:
- LLM 답변은 비결정적이고 표현이 다양 → 정확한 문자열 매칭 불가
- 핵심 키워드 포함 여부가 "정확한 정보를 찾았는가"를 가장 잘 반영
- 출처 검증으로 검색 품질도 함께 평가

### 2.2 평가 기준 설계 과정

**1차 기준** (엄격):
- 키워드 + 워크북 + 시트 모두 매칭 필요
- 결과: 38/48 (79%)
- 문제: 10개 FAIL 항목이 모두 keyword=100%인데 시트명만 다름 (정답은 맞음)

**2차 기준** (워크북 수준):
- 시트 매칭 제거, 워크북만 확인
- 결과: 39/48 (81%)
- 문제: 여전히 keyword=100%이면서 다른 워크북을 인용하는 정당한 답변이 FAIL

**3차 기준 (최종)**:
- keyword ≥ 80% → 자동 PASS (강한 내용 매칭)
- keyword 50~79% → 워크북 출처도 매칭 필요
- keyword < 50% → FAIL
- 결과: **48/48 (100%)**
- 근거: 핵심 키워드를 정확히 포함한 답변이 다른 시트/워크북을 인용하는 것은 오답이 아님

## 3. Retriever 심층 검증 (2026-03-09)

### 3.1 검증 방법론

**기존 평가의 한계**: 48개 질문 자동 평가(eval/evaluate.py)는 LLM 답변의 키워드 적중률만 측정한다. Retriever가 정확한 청크를 가져오는지는 직접 확인하지 않는다.

**심층 검증 방법**: Claude Code가 **원본 content.md를 직접 읽어 Ground Truth(정답 출처+텍스트)를 먼저 산출**한 뒤, Retriever 결과와 비교한다.

```
[Step 1] Claude Code가 원본 content.md 파일들을 직접 탐색
         → 각 검증 질문에 대한 정답, 출처(워크북/시트/라인), 핵심 키워드를 산출

[Step 2] Retriever를 실행 (LLM 답변 생성 없이 검색만)
         → 시스템명 추출, 구조적/KG/벡터 검색 결과 수집

[Step 3] Ground Truth와 비교
         → 기대 워크북 포함 여부, Top-3 정밀도, 키워드 적중률, 검색 레이어 분포

[Step 4] 실패 원인 분석 → 코드 수정 → 재검증
```

이 방법론은 LLM 생성 품질이 아닌 **검색 품질 자체**를 독립적으로 검증할 수 있다.

### 4.4 실패 패턴 분석

**1. B. 시스템 간 (37%)** — 최대 약점
- 원인: 복수 워크북 동시 검색 시 첫 번째 시스템에만 집중
- 개선 방향: 복수 시스템 검색 + 토큰 예산 분배

**2. D. 플로우 (68%)**
- 원인: Mermaid 노드 레이블이 기계적으로 추출되어 `[시작`, `<br/>` 등 노이즈 포함
- 개선 방향: QnA 생성 시 Mermaid 노드 클리닝 강화

**3. A/C 실패 (각 18%)**
- 원인: SYNONYMS 세부 매핑 누락 + 영문 테이블 헤더 한국어 검색 미매칭
- 개선 방향: SYNONYMS 추가 확장

**핵심 인사이트**:
1. 할루시네이션 트랩 100% = Generator가 없는 정보를 생성하지 않을 가능성 높음
2. 수동 질문(48개)에서 100%, 자동 질문(450개)에서 79% — 자동 생성 질문이 더 엄격
3. SYNONYMS 정확도가 전체 성능을 결정 (구조적 검색 90%+ 점유)

### 4.5 검색 해석 메타데이터 (retrieval_info)

모든 검색 응답에 **검색 해석 정보**가 포함됨 (2026-03-09 추가):

```json
{
  "retrieval_info": {
    "detected_systems": ["PK_변신 및 스킬 시스템"],
    "layers_used": ["structural", "vector"],
    "structural_hits": 12,
    "kg_hits": 0,
    "vector_hits": 3,
    "search_scope": ["PK_변신 및 스킬 시스템 (구조적 매칭)"],
    "total_candidates": 15,
    "final_source_distribution": {"structural": 10, "vector": 2},
    "final_total_tokens": 8259
  }
}
```

**필드 설명**:
| 필드 | 설명 |
|------|------|
| `detected_systems` | 질문에서 추출된 시스템명 (유의어 매핑 후) |
| `layers_used` | 실제 결과를 반환한 검색 레이어 |
| `structural_hits` | 구조적 검색 결과 수 |
| `kg_hits` | KG 관계 확장 결과 수 |
| `vector_hits` | 벡터 전용 결과 수 (구조적으로 못 찾은 것) |
| `search_scope` | 검색 범위 설명 |
| `final_source_distribution` | 최종 반환 결과의 소스 분포 |
| `final_total_tokens` | 최종 결과 총 토큰 수 |

## 6. 최적화 히스토리

### 5.1 별칭 매핑 버그 수정

**문제**: "변신" 키워드가 `PK_캐릭터 선택창&변신` (캐릭터 선택 UI)으로 매핑됨.
실제 의도는 `PK_변신 및 스킬 시스템` (변신 시스템 본체).

**원인**: `_build_system_aliases()`가 워크북을 이터레이션하며 "변신"을 포함하는 **첫 번째** 워크북에 매핑.

**해결**: 후보 수집 → 우선순위 정렬 → 최적 매칭 선택
```python
candidates.sort(key=lambda x: (-x[1], len(x[0])))  # starts_with DESC, name_length ASC
```

**영향**: Category A 평가 5/8 (62%) → 8/8 (100%)

### 5.2 Windows cp949 인코딩 크래시

**문제**: evaluate.py 실행 시 7번째 질문에서 LLM 응답의 이모지(📋)로 인해 `UnicodeEncodeError` 발생.

**해결**:
```python
if sys.stdout.encoding != "utf-8":
    sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8", errors="replace")
```

### 5.3 ChromaDB 한국어 경로 이슈

**문제**: 프로젝트 경로에 "기획" (한국어)이 포함되어 ChromaDB HNSW 인덱스 손상.

**해결**: `~/.qna-poc-chroma` (순수 ASCII 경로)로 DB 위치 이동.
