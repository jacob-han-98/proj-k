## 2. 검증 방법론: Human-in-the-Loop + Claude Code 반복 검증

### 2.1 왜 자동 검증이 아닌가

초기에는 "content.md에서 랜덤 항목을 뽑아 원본 이미지에 질의하는" 자동 검증을 계획했으나,
실제로는 **사람의 눈 + Claude Code의 분석 능력**을 결합한 반복 검증이 더 효과적이었다.

- 자동 검증: 정답/오답 판별은 가능하지만 **왜 틀렸는지, 어떻게 고칠지**를 알려주지 못함
- 반복 검증: 문제 발견 → 원인 분석 → 코드 수정 → 재실행 → 재검증 사이클이 즉시 가능

### 2.2 검증 프로세스

```
[1] 사용자: 원본 Excel 이미지와 content.md를 직접 대조
    ↓ 문제 발견 (누락, 오독, 중복, 할루시네이션 등)
[2] Claude Code: 중간 결과물 분석
    - _vision_output/merged.md (Vision 원본 해석)
    - _parse_ooxml_output/ (OOXML 보정 결과)
    - _final/content.md (최종 출력)
    ↓ 문제의 원인을 Stage별로 특정
[3] Claude Code: 원인에 맞는 수정 적용
    - Vision 프롬프트 개선 (할루시네이션, 누적 컨텍스트 조정)
    - Dedup 로직 추가/수정 (중복 패턴, false positive 방지)
    - OOXML 보정 강화 (엣지 감지, 색상 추출)
    - OCR 교정 규칙 조정 (changed_chars 임계값)
    ↓
[4] 재실행: 해당 시트만 --force로 재처리
    ↓
[5] 사용자 + Claude Code: 수정 결과 재검증
    ↓ 문제 해결 확인 or 새로운 문제 발견 → [1]로 반복
```

### 2.3 이 방식이 효과적인 이유

1. **Stage별 중간 결과물이 모두 보존됨**: Vision 원본, OOXML 보정, 최종 출력을 각각 비교 가능
2. **Claude Code가 코드와 결과를 동시에 분석**: 문제를 보고 → 코드에서 원인을 찾고 → 수정까지 1 사이클
3. **대표 파일 집중 검증**: PK_변신 및 스킬 시스템.xlsx (13시트, 다양한 콘텐츠 유형)로 모든 개선을 먼저 검증
4. **개선이 누적됨**: Dedup 2.0 → 2.1 → 2.5, 프롬프트 v1 → v11 등 반복 개선 이력이 코드에 반영

### 2.4 검증 사이클 실적

PK_변신 및 스킬 시스템.xlsx 기준 **5회 이상의 반복 검증 사이클** 수행:

| 사이클 | 발견된 문제 | 적용된 수정 | 결과 |
|--------|-----------|------------|------|
| 1 | 섹션 (5)~(8) 전체 누락 | Dedup metadata skip 로직 수정 | 해결 |
| 2 | 플로우차트 커넥터 누락 (합성→종료) | 크롭 재분석 + FLOWCHART_PROMPT v10 | 해결 |
| 3 | 빈 타일에서 할루시네이션 | is_blank_tile() + anti-hallucination 프롬프트 | 해결 |
| 4 | 타일 경계 중복 콘텐츠 잔존 | Dedup 7.5~8.6 단계 추가 (orphan, bold, annotation) | 대부분 해결 |
| 5 | OCR 오독 (알파>→알파2, 가체→개체) | OOXML 텍스트 코퍼스 + Sonnet OCR 교정 | 해결 |
| 6 | 등급 색상 부정확 (보라색→#7030A0) | OOXML 셀 배경색 추출 + 자동 교체 | 해결 |
| 7 | 병렬 실행 시 컨텍스트 오염 | 시트 간 병렬 + 타일 내 순차 구조로 분리 | 해결 |
